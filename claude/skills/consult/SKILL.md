# コンサルト

引数: $ARGUMENTS

## ペルソナ選択

ペルソナ一覧と思考特性は [references/personas.md](references/personas.md) を参照。

### 選択ルール

引数が空の場合は、利用可能なペルソナ一覧を表示してユーザーに選択を促す。

引数がある場合、以下の順で判定する:

1. 第1引数がペルソナ名と完全一致 → そのペルソナを選択、残りの引数をトピックとする
2. 第1引数がペルソナ名と部分一致（例: `arch` → `architect`）→ 該当ペルソナを選択。複数マッチした場合は候補を提示してユーザーに確認する
3. 一致するペルソナがない → 引数全体をトピックとみなし、内容から最適なペルソナを自動選択。選択理由を出力の冒頭に明示する

## 処理フロー

### Step 1: コンテキスト収集

- プロジェクトディレクトリ内にいる場合、技術スタックを確認（package.json、go.mod等）
- トピックが外部情報を必要とする場合は、WebSearchで最新の動向を1-2件検索し、分析の土台とする

### Step 2: ペルソナとしての分析

選択したペルソナの**専門性と思考様式**に徹底的になりきって分析する。各ペルソナの思考特性は [references/personas.md](references/personas.md) を参照。

### Step 3: 構造化出力

以下のフォーマットで報告する:

```
> ペルソナ: [ペルソナ名]（自動選択の場合は理由も記載）

## [ペルソナ名]としての分析: [トピック]

### 現状評価
[トピックの現状を、このペルソナの専門的視点で評価]
[プロジェクトのコードがある場合は、具体的なコードを参照して評価]

### 推奨事項
1. [最優先の推奨事項と根拠]
2. [次に重要な推奨事項]
3. ...

### リスクと注意点
- [このペルソナが特に重視する懸念点]
- [見落としがちだが重要なポイント]

### 次のステップ
- [ ] [すぐにできるアクション]
- [ ] [中期的に取り組むべき項目]
- [ ] [長期的に検討すべき項目]

### 参考情報
- [関連するリソース、ドキュメント、ツール]
```

日本語で結果を報告してください。
